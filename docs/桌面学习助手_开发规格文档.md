## 1. 项目概述

### 1.1 产品定位
桌面学习助手"阿米娅（Amiya）"是一款面向学生的AI语音陪伴设备，以大模型语音交互为核心，集成手机收纳管理、坐姿监测、走神追踪等功能，帮助用户建立专注的学习环境。

### 1.2 核心功能模块

| 模块 | 功能描述 | 优先级 |
|------|---------|--------|
| 语音助手 | 大模型语音交互（中英双语），角色扮演Amiya性格 | P0 |
| 手机管理器 | 语音指令进入专注模式，红外检测+舵机控制盒盖开关 | P0 |
| 坐姿端正器 | TOF距离监测，过近时语音提醒 | P0 |
| 走神监测 | 二维云台+摄像头人脸追踪，离开画面后智能处理 | P0 |
| 对话调度 | 上下文管理、长记忆存储、对话流畅性保障（M6+ 规划） | P0 |
| LED指示 | 多色LED显示系统当前状态 | P1 |
| 心率血氧 | 脉搏血氧检测（后续扩展） | P2 |

### 1.3 技术栈总览

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 大模型服务 | 阿里云百炼API | LLM(qwen-plus) + ASR(Paraformer-Realtime-V2) + TTS(cosyvoice-v3-flash) 统一接入 |
| 语音前端 | webrtcvad + 流式ASR关键词检测 | VAD语音活动检测 + Paraformer-Realtime-V2流式唤醒 |
| 视觉处理 | OpenCV (Python) | 人脸检测与跟踪 |
| 云台控制 | PID控制器 | 自定义实现 |
| 外设驱动 | GPIOZero | 树莓派GPIO控制 |
| 音频IO | PyAudio + USB声卡 | 音频采集与播放 |
| 数据存储 | JSON文件 | 长期记忆与配置存储 |
| 进程通信 | multiprocessing.Queue | 跨模块消息传递 |
| 部署方式 | systemd服务 | 开机自启动守护进程 |

## 2. 系统架构设计

### 2.1 进程架构（多模块并行）

系统采用 **1个主进程 + 3个子进程** 的架构（M8已完成）

> 多进程架构已通过 `MessageBus` + `IPCMessage`（`src/message_bus.py`）实现。传感器进程入口为 `sensor_process_loop()`（`src/processes/sensor_process.py`）。
，通过 `multiprocessing.Queue` 进行进程间通信（IPC）。

```
+--------------------------------------------------+
|                   主进程 (MainProcess)            |
|  +--------------------------------------------+  |
|  |           对话调度器 (DialogManager)        |  |
|  |  - VAD语音检测 + 流式ASR关键词唤醒 + ASR |  |
|  |  - 大模型对话 + 函数调用 + TTS             |  |
|  |  - 上下文管理 + 长期记忆读写               |  |
|  |  - 各功能模块的状态协调                    |  |
|  +--------------------------------------------+  |
|                      |                           |
|  +--------------------------------------------+  |
|  |         状态机控制器 (StateController)      |  |
|  |  - 专注模式状态机执行                      |  |
|  |  - 各模块优先级仲裁                        |  |
|  |  - LED状态指示更新                         |  |
|  +--------------------------------------------+  |
+--------------------------------------------------+
          | Queue            | Queue            | Queue
+---------v---------+ +------v-------+ +--------v--------+
| 视觉子进程        | | 传感器子进程 | | 外设控制子进程  |
| (VisionProcess)  | | (SensorProc) | | (DeviceProcess) |
| - 人脸检测        | | - TOF测距    | | - 舵机控制      |
| - 云台PID控制     | | - 红外检测   | | - LED控制      |
| - 人脸坐标输出    | | - 血氧读取   | | - 蜂鸣器(预留) |
+-------------------+ +--------------+-------------------+
```

### 2.2 模块职责划分

| 进程 | 职责 | 运行方式 |
|------|------|----------|
| 主进程-对话调度器 | 音频采集、唤醒检测、ASR、LLM对话、TTS播放、记忆管理 | asyncio + threading |
| 主进程-状态机控制器 | 专注模式状态流转、模块优先级裁决、LED状态映射 | asyncio |
| 视觉子进程 | 摄像头帧捕获、人脸检测、PID计算、PWM输出到舵机 | 独立Process |
| 传感器子进程 | TOF距离读取、红外传感器读取、血氧读取（预留） | 独立Process |
| 外设控制子进程 | 手机盒舵机控制、LED控制、接收指令并执行 | 独立Process |

### 2.3 通信协议（Queue消息格式）

所有进程间消息采用统一字典格式：

```python
{
    "type": "消息类型",           # 必填，见下表
    "source": "发送方",          # 必填：main / vision / sensor / device
    "target": "接收方",          # 可选，broadcast表示广播
    "timestamp": 1714521600.0,   # 时间戳
    "payload": {}                # 具体数据内容
}
```

| 消息类型 (type) | 发送方 | 接收方 | 说明 |
|----------------|--------|--------|------|
| `face_detected` | vision | main | 人脸检测结果（位置/大小/置信度） |
| `face_lost` | vision | main | 人脸丢失超阈值 |
| `servo_angle` | vision | device | 云台舵机角度更新 |
| `distance_tof` | sensor | main | TOF距离测量值 |
| `phone_detected` | sensor | main | 红外检测到手机放入 |
| `phone_removed` | sensor | main | 红外检测到手机取出 |
| `focus_command` | main | device | 专注模式指令（开盖/关盖） |
| `led_state` | main | device | LED状态更新 |
| `user_speech` | main | main | ASR识别结果（内部流转） |
| `tts_request` | main | main | TTS播放请求（内部流转） |
| `system_event` | any | main | 系统级事件（错误/状态变更） |

### 2.4 主循环时序

```
[主进程循环 - 每50ms迭代]
 1. VAD持续监听麦克风，检测到语音活动后启动流式ASR
 2. ASR流式结果中检查是否含唤醒词"阿米娅"/"Amiya"
 3. 若唤醒成功：启动VAD采集完整语音 → ASR → LLM → TTS → 播放
    - 收到 face_lost → 触发走神处理流程
    - 收到 distance_tof < THRESHOLD → 触发坐姿提醒
    - 收到 phone_detected/removed → 更新专注模式状态机
 4. 执行状态机控制器逻辑（专注模式计时、状态转移）
 5. 更新LED状态
```

## 3. 唤醒检测与VAD方案

### 3.1 唤醒词检测（ASR-Based流式关键词检测）

由于中国大陆网络环境限制，本项目**不采用 Picovoice/Porcupine 等需要海外注册的服务**。唤醒词检测使用纯本地 VAD + Paraformer-Realtime-V2 流式识别实现。

**方案原理：**

```
[待机状态 - 持续VAD监听]
     ↓ VAD检测到连续语音活动（≥300ms）
[唤醒确认 - 启动流式ASR]
     ↓ 边采集边送入Paraformer-Realtime-V2
[实时文本匹配]
     ├─ 文本含"阿米娅"/"Amiya" → 唤醒成功
     └─ 文本不含唤醒词 → 丢弃，回到待机
     ↓ 唤醒成功
[继续采集后续语音] → 作为用户完整输入 → LLM → TTS
```

| 项目 | 方案 |
|------|------|
| 检测方式 | **webrtcvad** 本地检测语音活动 |
| 唤醒确认 | **Paraformer-Realtime-V2** 流式识别（百炼API） |
| 支持唤醒词 | "阿米娅" / "Amiya" / "amiya"（ASR语义理解，不区分大小写） |
| 唤醒词容错 | ASR识别结果包含唤醒词即可（如"阿米娅在吗"也能触发） |
| 输入音频 | 16kHz采样率，16bit，单声道 |
| 检测延迟 | 约0.5-1.5秒（取决于说话速度和网络状况） |
| 运行方式 | 主进程中独立线程持续监听麦克风 |

**流式唤醒检测详细流程：**

```
[持续麦克风监听]
     ↓
webrtcvad 逐帧检测（30ms/帧）
     ↓ 连续3帧（90ms）检测到语音
启动音频环形缓冲区（缓存最近2秒音频）
     ↓
同步启动Paraformer-Realtime-V2流式识别
     ↓
实时接收ASR中间结果（每300ms返回一次）
     ↓ 任意中间/最终结果包含"阿米娅"或"Amiya"
确认唤醒成功！停止本次唤醒检测
     ↓
播放提示音（可选）
     ↓
继续采集后续音频（同一条流式连接）直到VAD检测到语音结束
     ↓
获取完整ASR结果 → 送入LLM
     ↓
LLM → TTS → 播放回复
```

**唤醒与对话复用同一条ASR连接：**

Paraformer-Realtime-V2 支持流式识别。唤醒检测和后续对话采集可以复用同一个流式连接：
1. VAD检测到语音 → 启动流式ASR连接
2. ASR流式结果中检查唤醒词
3. 检测到唤醒词后，**不关闭ASR连接**，继续采集同一条音频流
4. VAD检测到语音结束后，取完整ASR结果中的**唤醒词之后的内容**作为用户输入

这样整个唤醒+对话过程只调用一次ASR API，最大程度降低成本。

**误触发控制：**

| 防护机制 | 说明 |
|----------|------|
| VAD前置过滤 | 只有检测到有效语音（≥300ms）才启动ASR，环境噪声不触发 |
| 静音丢弃 | ASR识别结果为纯标点/空文本 → 立即丢弃回到待机 |
| 非唤醒词丢弃 | 文本不含"阿米娅"/"Amiya" → 丢弃，累计3次后暂停监听2秒 |
| 唤醒冷却 | 一次唤醒后5秒内不再重复触发（防止同一句话重复唤醒） |

**Mock模式下的唤醒测试：**

Mock模式下，不启动真实ASR唤醒检测，改为：
- 终端输入 `"阿米娅"` 或 `"Amiya"` 模拟唤醒词触发
- 或直接按 `Enter` 键触发（方便调试）

### 3.2 VAD（语音活动检测）

| 项目 | 方案 |
|------|------|
| 库 | **webrtcvad** (Google WebRTC VAD) |
| 模式 |  aggressiveness=2（中等敏感度） |
| 帧长 | 30ms (480 samples @ 16kHz) |
| 语音结束判定 | 连续15帧（450ms）无语音活动 |
| 最大采集时长 | 15秒（超时强制截断） |
| 最小有效语音 | 500ms（低于此长度视为噪声，丢弃） |

**VAD在唤醒与对话中的应用：**
- **待机唤醒阶段**：VAD持续监听麦克风，检测到语音活动后启动流式ASR做唤醒确认
- **首轮唤醒后**：同一条ASR连接继续采集，VAD判断语音结束点，获取完整语句
- **连续对话中**：VAD检测用户插话/新语句，自动开启新采集
- **等待播放完毕**：TTS播放期间等待音频输出完成，播放完毕后自动进入下一轮录音采集

## 4. 语音交互流程设计

### 4.1 正常对话时序

```
[待机状态 - VAD持续监听麦克风]
     ↓ VAD检测到语音活动 + ASR流式识别确认含"阿米娅"
[唤醒成功 - 播放简短提示音]
     ↓ VAD检测到用户语音（同一条ASR流继续采集）
[聆听状态 - 采集完整语句]
     ↓ 语音结束
[ASR识别 - Paraformer-Realtime-V2流式识别（百炼API）]
     ↓ 文本结果
[LLM处理 - 百炼qwen-plus 上下文注入 + 函数调用判断]
     ├─ 需要调用外设 → 执行函数 → 获取结果 → LLM再处理
     └─ 纯对话 → 直接生成回复
     ↓
[TTS合成 - 百炼cosyvoice-v3-flash]
     ↓
[音频播放 - PyAudio输出到USB声卡（重采样16kHz→24kHz）]
     ↓
[等待下一句 / 超时返回待机]
```

### 4.2 连续对话机制

| 场景 | 处理策略 |
|------|---------|
| 对话间隙 | 保持唤醒状态，无需重复唤醒词可直接说话 |
| 对话超时（10秒） | 自动返回待机，需再次说唤醒词唤醒 |
| 连续2轮无声 | 语音提示用户，返回待机 |
| 误唤醒（非唤醒词语音） | ASR结果不含"阿米娅"/"Amiya" → 丢弃音频，自动返回待机 |
| 用户长时间沉默 | TTS播放后等待录音（最多8秒），超时后返回待机 |

### 4.3 函数调用（Tool Use）设计

大模型支持以下可调用的工具函数：

```yaml
tools:
  - name: "set_focus_mode"
    description: "开启专注模式，管理手机盒"
    parameters:
      duration_minutes: int   # 专注时长（分钟）
    returns: "success/failed + 当前专注状态"

  - name: "open_phone_box"
    description: "打开或临时打开手机盒"
    parameters:
      reason: str            # 原因：temporary（临时拿手机/暂停专注）
    returns: "success/failed"

  - name: "get_focus_status"
    description: "查询当前专注模式状态"
    returns: "状态/剩余时间/是否暂停"

  - name: "end_focus_mode"
    description: "结束当前专注模式（彻底结束，非临时拿手机）"
    returns: "success/failed"

  - name: "set_user_nickname"
    description: "设置/修改用户称呼"
    parameters:
      nickname: str
    returns: "success + 确认语句"
```

## 5. 对话上下文管理策略

> **注意**：本章描述 M6 实现的记忆架构。对话历史由 `src/dialog_manager.py` 管理，跨会话记忆与上下文压缩由 `src/memory_manager.py` 实现（含摘要持久化、用户昵称/偏好记忆）。

### 5.1 三层记忆架构

```
+-------------------------------------------------+
|               实时对话上下文                      |
|  (当前会话最近10轮完整对话，约2000 tokens)       |
|  - 存储：内存变量                                |
|  - 作用：支撑当前对话的连贯性                    |
+-------------------------------------------------+
                      ↓ 会话结束 / 超过10轮
+-------------------------------------------------+
|               短期记忆摘要                        |
|  (本日对话关键摘要，最多保留近5次会话摘要)       |
|  - 存储：内存 + memory_today.json               |
|  - 触发：每轮对话后LLM自动提取关键信息           |
+-------------------------------------------------+
                      ↓ 每日整理 / 会话数超5
+-------------------------------------------------+
|               长期记忆存储                        |
|  (用户偏好、习惯、常见话题、高频问题)            |
|  - 存储：longterm_memory.json                   |
|  - 触发：每日首次启动时整理，或手动触发          |
+-------------------------------------------------+
```

### 5.2 实时上下文管理

| 参数 | 设定值 | 说明 |
|------|--------|------|
| 最大保留轮数 | 10轮（20条消息：用户+助手各10） | 超出后触发摘要 |
| Token上限 | 3000 tokens | 达到80%（2400）时提前触发摘要 |
| 摘要触发 | 超出轮数 或 接近token上限 | 异步执行，不阻塞对话 |
| 摘要方式 | 调用LLM对当前对话生成200字摘要 | 提取关键信息、用户意图、约定事项 |

### 5.3 上下文压缩策略

当对话轮数超过10轮时，执行以下压缩流程：

```
当前会话：第1轮 ~ 第10轮
     ↓
调用LLM生成摘要（保留第1-5轮摘要 + 第6-10轮完整对话）
     ↓
新结构：[摘要_1_5] + [完整对话_6_10] + [新对话_11...]
     ↓
当第11-15轮又满了，再次压缩：[摘要_1_10] + [完整对话_11_15]
```

**摘要模板：**
```
[对话摘要] 本次对话中用户询问了{topic}，我回答了{summary}。
用户偏好/约定：{user_preferences}。
待办/待确认事项：{action_items}。
```

### 5.4 多会话记忆融合

- 每次启动时，读取 `memory_today.json` 中的近期会话摘要
- 将摘要注入系统提示词（system prompt），让LLM了解近期上下文
- 每日凌晨4点（或设备长期闲置后）触发长期记忆整理

### 7.3 Paraformer-Realtime-V2 接入说明

ASR通过阿里云百炼API流式调用，与LLM、TTS共享同一API Key。

**Paraformer-Realtime-V2 特点：**
- **流式实时识别**：边说话边返回识别结果，无需等待整句说完
- **低延迟**：首字返回延迟 < 300ms，适合对话场景
- **中英混合支持**：自动识别中文、英文及混合内容
- **抗噪能力强**：针对室内环境优化，适应桌面学习场景
- **长音频支持**：单次最长支持60秒连续语音识别

**接入方式（流式）：**

```python
import dashscope
from dashscope.audio.asr import Recognition

# 初始化流式识别（通过websocket）
recognition = Recognition(
    model="paraformer-realtime-v2",
    format="pcm",
    sample_rate=16000,
    api_key=os.getenv("ALIBABA_API_KEY")
)

# 流式回调处理
def on_result(result):
    if result.get("is_final"):
        text = result.get("text", "")
        # 送入对话调度器处理
        dialog_manager.handle_user_speech(text)

# 启动识别
recognition.start(callback=on_result)

# VAD采集的音频chunk实时送入
while recording:
    chunk = audio_buffer.get_chunk()
    recognition.send_audio(chunk)
    
# 语音结束，通知识别完成
recognition.stop()
```

**与VAD的配合流程：**

```
唤醒词触发
    ↓
启动VAD + 启动Paraformer-Realtime-V2流式识别
    ↓
音频chunk实时送入API ──────→ 实时返回中间识别结果（显示/丢弃）
    ↓
VAD检测到语音结束
    ↓
收取API最终识别结果（is_final=true）
    ↓
最终文本 → LLM → TTS
```

**容错处理：**
- API请求失败时自动重试2次，仍失败则语音提示"没听清，请再说一遍"
- 网络抖动导致识别中断时，已返回的部分结果若置信度足够高，仍可使用

### 8.1 视觉模块参数

| 参数 | 设定值 |
|------|--------|

| 置信度阈值 | 0.7 |
| 画面中心死区 | ±40像素（避免微小抖动导致舵机频繁运动） |

### 8.2 PID控制器设计

二维云台使用独立的PID控制器分别控制 **水平（Pan）** 和 **俯仰（Tilt）** 轴。

```python
class PIDController:
    def __init__(self, Kp, Ki, Kd, output_limit=(-45, 45)):
        self.Kp = Kp
        self.Ki = Ki
        self.Kd = Kd
        self.integral = 0
        self.prev_error = 0
        self.output_limit = output_limit
        self.integral_limit = 20  # 积分限幅，防止windup

    def update(self, error, dt):
        # 误差 = 人脸中心坐标 - 画面中心坐标
        self.integral += error * dt
        self.integral = max(min(self.integral, self.integral_limit), -self.integral_limit)
        derivative = (error - self.prev_error) / dt
        output = self.Kp * error + self.Ki * self.integral + self.Kd * derivative
        self.prev_error = error
        return max(min(output, self.output_limit[1]), self.output_limit[0])
```

**PID参数（初始值，需实际调参）：**

| 轴 | Kp | Ki | Kd | 说明 |
|----|----|----|----|------|
| Pan（水平） | 0.08 | 0.01 | 0.02 | 响应稍慢但平滑 |
| Tilt（俯仰） | 0.06 | 0.008 | 0.015 | 俯仰移动幅度通常更小 |

### 8.3 人脸追踪算法流程

```
[每帧循环 @ 15FPS]
     ↓
捕获帧 → 640x480
     ↓
每3帧执行人脸检测？ → 是：Haar Cascade检测 → 获取人脸bbox
                   → 否：跳过检测（无预测，当前未实现卡尔曼滤波）
     ↓
计算人脸中心与画面中心偏差 (dx, dy)
     ↓
死区判断：|dx| < 40 且 |dy| < 30？ → 是：跳过PID更新
                                    → 否：继续
     ↓
PID计算目标角度增量
     ↓
角度限幅检查：Pan ±90°，Tilt ±45°
     ↓
发送角度指令到外设控制子进程 → 输出PWM
     ↓
[主进程消息] 定期发送 face_detected / face_lost
```


## 9. 专注模式状态机设计

### 9.1 状态定义

```
                    +------------------+
                    |    IDLE(空闲)    |

                             | 用户语音"开启专注模式" / set_focus_mode()
                             v
              +--------------+---------------+
              |   WAITING_PHONE(等待放手机)  |
              |   语音引导：请把手机放入盒子  |
              |   LED：蓝色呼吸灯              |
              +--------------+---------------+
                             | 红外传感器检测到手机放入
                             v
              +--------------+---------------+
              |   BOX_CLOSED(盒盖已关闭)     |
              |   调用舵机关闭盒盖             |
              |   LED：绿色常亮               |
              +--------------+---------------+
                             |
                             v
         +-------------------+-------------------+
         |          FOCUSING(专注中)               |
         |   倒计时运行，每分钟语音提示剩余时间      |
         |   LED：绿色呼吸灯（频率随时间递减）      |
         |   坐姿监测和走神监测正常运行              |
         +-------------------+-------------------+
                             |
            +----------------+----------------+
            |                                 |
            v                                 v
   +--------+---------+            +----------+----------+
   | PAUSED(已暂停)    |            | COMPLETED(已完成)   |
   | 临时开盒状态       |            | 倒计时归零           |
   | LED：黄色常亮      |            | 语音恭喜+开盖        |
   +--------+---------+            | LED：彩虹闪烁→空闲   |
            |                      +---------------------+
            | 用户说"放回去了" /
            | 红外检测到手机放入
            v
   回到 FOCUSING 状态，恢复计时
```

> **已实现**：`StateController`（`src/state_controller.py`）管理 `FocusState` 枚举的 6 个状态，集成于 `ToolExecutor.state_ctrl` 属性。

### 9.2 状态转移表

| 当前状态 | 触发事件 | 下一状态 | 动作 |
|----------|----------|----------|------|
| IDLE | 语音指令"我要专注N分钟"（默认40分钟） | WAITING_PHONE | 设置目标时长，语音引导放手机 |
| WAITING_PHONE | 红外检测到手机 | BOX_CLOSED | 关盖，启动倒计时 |
| WAITING_PHONE | 超时60秒未放手机 | IDLE | 语音提醒"下次再专注吧"，返回空闲 |
| BOX_CLOSED | 盒盖关闭完成 | FOCUSING | 正式开始专注计时 |
| FOCUSING | 用户说"我要接电话" | PAUSED | 暂停计时，开盖 |
| FOCUSING | 倒计时归零 | COMPLETED | 语音祝贺，开盖，统计数据 |
| FOCUSING | 用户说"取消专注模式" | IDLE | 确认后开盖，取消计时 |
| PAUSED | 红外检测到手机放入 | FOCUSING | 关盖，恢复计时 |
| PAUSED | 暂停超10分钟未放回 | IDLE | 语音提醒"专注模式已取消" |
| COMPLETED | 开盖完成 | IDLE | 记录专注数据到长期记忆 |

### 9.3 专注模式计时规则

- **基础计时单位**：秒级（使用 `time.monotonic()` 保证不受系统时间调整影响）
- **暂停逻辑**：临时开盒期间计时器暂停，放入后从暂停点恢复
- **语音提醒策略**：
  - 每10分钟播报一次剩余时间（如"还剩15分钟，{nickname}继续加油"）
  - 剩余5分钟时提醒一次
  - 剩余1分钟时提醒一次
  - 完成时播放祝贺语音
- **最小时长**：5分钟
- **最大时长**：120分钟
- **默认时长**：40分钟

### 9.4 临时开盒的语音指令集

用户可在专注过程中通过语音要求临时开盒：

| 用户说法示例 | 系统响应 |
|-------------|---------|
| "我要接电话" | "好的，{nickname}，接完电话记得放回来继续专注哦" → PAUSED |
| "我要回消息" | 同上 |
| "我要查个资料" | "好的，尽快回来哦" → PAUSED |
| "我有急事" | "好的，专注模式我先暂停了" → PAUSED |

**安全机制**：临时开盒后10分钟未放回 → 自动取消专注模式，进入IDLE。

## 14. 异常处理与降级策略

### 14.1 异常分类与处理

| 异常类型 | 检测方式 | 降级策略 | LED提示 |
|----------|---------|---------|---------|
| 网络断开 | API请求超时/失败 | 语音提示"网络好像不太稳定"，本地缓存最近5条TTS音频作为应急回复，网络恢复后自动恢复 | 红色慢闪 |
| API Key失效 | HTTP 401/403 | 语音提示"服务暂时不可用"，进入本地简单模式（预置语音回复） | 红色快闪 |
| 摄像头故障 | OpenCV无法打开设备 | 人脸追踪功能关闭，走神监测降级为纯时间提醒（每20分钟提醒一次），其他功能正常 | 橙色常亮 |
| TOF传感器故障 | I2C读取失败/超范围值 | 坐姿监测功能关闭，语音告知用户"坐姿检测暂时不可用" | 黄色快闪1次 |
| 红外传感器故障 | 信号异常抖动 | 专注模式改为纯手动控制（语音确认开关盖），禁用自动检测 | 黄色快闪1次 |
| 舵机故障 | PWM无响应/电流异常 | 相应舵机功能关闭，语音提示用户手动操作 | 红色快闪2次 |
| USB声卡断开 | PyAudio设备枚举失败 | 系统进入静音模式，LED以特定模式提示，尝试每10秒重新初始化音频 | 紫色快闪 |
| 树莓派过热 | CPU温度 > 80°C | 降低视觉模块帧率到5FPS，降低TTS采样率，语音提醒用户注意散热 | 红色呼吸灯 |

## 17. Mock开发标准

### 17.1 Mock原则

阶段1在PC上开发时，所有硬件交互通过Mock层实现：

1. **接口一致性**：Mock类与真实驱动类实现完全相同的公共接口
2. **行为模拟**：Mock能模拟硬件的主要行为（延迟、返回值范围、偶发异常）
3. **可配置性**：通过配置文件切换Mock/真实模式，无需修改业务代码
4. **日志记录**：Mock操作打印到日志，便于调试

### 附录D：风险评估与应对

| 风险项 | 影响 | 应对方案 |
|--------|------|---------|
| 树莓派5性能不足 | 语音延迟高、人脸追踪卡顿 | 阶段1先在PC充分优化；必要时降低视觉帧率或使用模型量化 |
| 阿里云API延迟/不稳定 | 对话卡顿 | 实现请求重试+超时降级；缓存常用TTS音频 |
| ASR流式唤醒延迟偏高 | 网络波动导致唤醒确认慢 | 优化VAD灵敏度减少无效触发；本地缓存网络良好时预连接ASR WebSocket；Mock模式下用本地输入模拟 |
| USB声卡兼容性 | 音频无法正常工作 | 准备2-3款不同USB声卡备选；支持蓝牙音频fallback |
| 舵机精度不足 | 盒盖无法紧密关闭 | 选用大扭矩舵机；软件层面增加位置校准 |
| 开发进度延误 | 里程碑无法按时完成 | 阶段1优先保证语音交互+专注模式；坐姿和走神可后续迭代 |
