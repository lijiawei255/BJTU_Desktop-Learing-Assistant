> BJTU 桌面学习助手（Amiya）项目协作开发指南
> 适用阶段：阶段一（PC 模拟开发）及后续全部阶段

## 一、初始搭建（只做一次）

### 1.1 安装 Anaconda3

前往 [anaconda.com/download](https://www.anaconda.com/download) 下载 Windows 版并安装。

> 如已安装 Anaconda3，可跳过此步。终端输入 `conda --version` 确认可用。

### 1.2 获取阿里云百炼 API Key

1. 访问 [阿里云百炼控制台](https://bailian.console.aliyun.com/)
2. 登录后进入「模型服务」→「API Key 管理」
3. 创建新的 API Key 并复制保存

> 项目使用千问系列模型（LLM + ASR + TTS），统一通过百炼 API 接入。API Key 是调用这些服务的凭证，每人需用自己的 Key。

### 1.3 克隆仓库

```bash
git clone https://github.com/lijiawei255/BJTU_Desktop-Learing-Assistant.git
cd BJTU_Desktop-Learing-Assistant
```

### 1.4 一键创建 Conda 环境

```bash
conda env create -f environment.yml
conda activate amiya
```

> `environment.yml` 定义了与主开发者完全一致的 Python 3.11 环境及所有依赖包。创建过程约 2-5 分钟。

### 1.5 配置 API Key

```bash
cp .env.example .env
```

用文本编辑器打开 `.env`，将 `sk-your-api-key-here` 替换为步骤 1.2 中获取的百炼 API Key：

```
ALIBABA_API_KEY=sk-你的真实Key
```

> `.env` 已被 `.gitignore` 排除，不会被提交到 GitHub。**请勿将 API Key 分享给他人或写到代码中。**

### 1.6 验证环境

```bash
python -c "import webrtcvad, pyaudio, numpy, cv2, dashscope, pydantic; print('OK')"
```

输出 `OK` 即表示环境搭建成功，可以开始开发。

## 二、日常开发流程

### 2.1 分支模型

```
main           ← 稳定版本，不直接提交
develop        ← 日常集成分支（所有 feature 分支向这里合并）
feature/xxx    ← 各自的功能分支，从 develop 拉出
```

### 2.2 每次开发前的准备

```bash
# 1. 切回 develop 分支
git checkout develop

# 2. 拉取最新代码（同步协作方的变更）
git pull origin develop

# 3. 从最新 develop 拉出自己的功能分支
git checkout -b feature/你的功能名
```

> 功能分支命名示例：`feature/config-system`、`feature/audio-pipeline`、`feature/focus-state-machine`

### 2.3 开发中

在功能分支上正常编写代码。如果当天无法完成，记得在下班前 `git push origin feature/xxx` 推送到远程，避免本地代码丢失。

### 2.4 提交代码

```bash
# 查看变更
git status

# 暂存相关文件
git add <文件1> <文件2> ...

# 提交（使用约定前缀）
git commit -m "feat: 实现了xxx功能"

# 推送到 GitHub
git push origin feature/你的功能名
```

### 2.5 提交信息前缀约定

| 前缀 | 用途 | 示例 |
|------|------|------|
| `feat:` | 新功能 | `feat: 实现配置管理模块` |
| `fix:` | 修复 Bug | `fix: 修复 ASR 超时未重试的问题` |
| `docs:` | 文档变更 | `docs: 更新协作者方法文档` |
| `test:` | 测试相关 | `test: 添加状态机单元测试` |
| `mock:` | Mock 相关 | `mock: 新增摄像头 Mock 类` |
| `refactor:` | 重构 | `refactor: 提取消息总线为独立模块` |

### 2.6 提交 Pull Request（PR）

1. 在 GitHub 上进入仓库页面
2. 从你的 `feature/xxx` 分支向 `develop` 分支创建 Pull Request
3. 填写 PR 标题和描述，说明改了什么、为什么要这样改
4. 通知协作者 Review
5. Review 通过后，由 PR 发起者点击「Merge pull request」合并

### 2.7 PR 合并后

```bash
# 切回 develop 并拉取合并后的最新代码
git checkout develop
git pull origin develop

# 删除本地已完成的功能分支（可选）
git branch -d feature/你的功能名
```

## 三、环境同步

### 3.1 当有人修改了依赖

如果项目的 `environment.yml` 或 `requirements.txt` 发生了变更（比如新增了第三方包），拉取最新代码后需要同步本地环境：

```bash
# 同步 conda 环境变更
conda env update -f environment.yml --prune

# 同步 pip 依赖变更
pip install -r requirements.txt
```

### 3.2 如果你新增了依赖

在提交代码前，务必同步更新以下文件：

| 文件 | 更新内容 |
|------|---------|
| `environment.yml` | `pip:` 段中添加新包及版本号 |
| `requirements.txt` | 添加新包及最低版本要求 |

> 原则：让协作者 `pull` 代码后只需执行 3.1 节的两条命令即可保持环境一致。

## 四、常见问题

### Q1: `conda: command not found`

Anaconda 未安装或未添加到系统 PATH。重新安装 Anaconda3 时勾选「Add Anaconda to my PATH environment variable」。

### Q2: `conda env create` 很慢

conda 默认源在海外，可配置国内镜像加速：

```bash
conda config --add channels https://mirrors.tuna.tsinghua.edu.cn/anaconda/pkgs/main/
conda config --add channels https://mirrors.tuna.tsinghua.edu.cn/anaconda/cloud/conda-forge/
conda config --set show_channel_urls yes
```

### Q3: `import webrtcvad` 报 `ModuleNotFoundError: No module named 'pkg_resources'`

webrtcvad 与新版本 setuptools 不兼容。手动降级 setuptools：

```bash
pip install "setuptools>=65.0,<70.0"
```

### Q4: 我的 `.env` 文件没生效

确认文件名是 `.env`（不是 `.env.example` 或 `.env.txt`），且文件位于项目根目录。

### Q5: 冲突了怎么办？

```bash
# 在 develop 分支上拉取最新代码
git checkout develop
git pull origin develop

# 切回你的 feature 分支，将 develop 合并进来
git checkout feature/xxx
git merge develop

# 手动解决冲突文件中的 <<<<<< 标记
# 解决后：
git add <冲突文件>
git commit -m "merge: 合并 develop 解决冲突"
git push origin feature/xxx
```
