# 业务逻辑说明

本文档描述阿米娅（Amiya）桌面学习助手当前已实现的完整业务逻辑，供协作者快速理解项目并开展后续开发。

---

## 一、系统总体架构

```
┌─────────────────────────────────────────────────────────┐
│                      主循环 (main.py)                    │
│  IDLE → AWAKE → LISTENING → PROCESSING → WAITING → ...  │
├─────────────────────────────────────────────────────────┤
│  语音管道: AudioHandler → VAD → WakeWord → ASR → LLM → TTS │
│  工具系统: LLM Function Calling → ToolExecutor → Devices  │
│  硬件抽象: Device Manager → Mock (PC) / Real (RPi)       │
└─────────────────────────────────────────────────────────┘
```

**运行模式**：通过 `data/config.json` 的 `mock` 字段逐设备控制。PC 开发默认全 Mock，树莓派部署设 `"enabled": false` 切换到真实驱动。

## 二、语音交互逻辑

### 2.1 状态流转

```
                    ┌──────────┐
                    │   IDLE   │ 等待唤醒词
                    └────┬─────┘
                    唤醒词检测到 "阿米娅" / "Amiya"
                         │
                    ┌────▼─────┐
                    │  AWAKE   │ TTS: "嗯，我在听，请说。"
                    └────┬─────┘
                         │
              ┌──────────┼──────────┐
              ▼          ▼          ▼
         LISTENING   PROCESSING   WAITING
         (录音采集)  (ASR→LLM→TTS) (播放完毕等下一轮)
              │          │          │
              └──────────┴──────────┘
                         │
              超时/静默/ASR失败过多
                         │
                    ┌────▼─────┐
                    │   IDLE   │ TTS: "有需要的时候，再呼唤阿米娅..."
                    └──────────┘
```

### 2.2 唤醒词检测

1. VAD 检测到语音活动 → 累积音频
2. 每约 1.5 秒将累积音频发送至阿里云百炼 ASR（流式）
3. ASR 返回文本包含 "阿米娅" 或 "Amiya" → 唤醒确认
4. 进入 5 秒冷却期，期间不再响应唤醒词

### 2.3 多轮对话

**正常流程**：
1. `AudioHandler.record_until_silence()` — VAD 检测语音结束后停止录音
2. `ASRClient.recognize_once()` — 发送至云端 Paraformer-Realtime 识别
3. 过滤：过短文本（<2 字符）丢弃
4. `LLMClient.stream_chat()` — 流式请求 qwen-plus，逐 token 回调
5. `SentenceSplitter` 按标点分割句子 → 逐句送入 `TTSClient`
6. TTS 后台播放线程排队播放 cosyvoice-v3-flash 合成语音
7. 等待播放完毕 → 回到 LISTENING 继续下一轮

**超时与退出**：
- 6 秒无对话 → TTS 提示后返回 IDLE（`conversation_timeout_seconds=6`）
- 连续 2 次空录音 → TTS 提示后返回 IDLE
- ASR 连续 5 次失败 → 降级模式，TTS 提示后返回 IDLE
- LLM 连续 5 次失败 → 降级模式，TTS 提示后返回 IDLE
- 打断功能已禁用（`barge_in_enabled=false`）

**[SKIP] 标记**：LLM 判断用户消息不是对自己说的（背景闲聊/噪音），回复 `[SKIP]` → 过滤该轮对话，不播放任何语音。

**[EXIT] 标记**：LLM 判断用户要结束对话（"再见"、"去忙了"），回复末尾附加 `[EXIT]` → 退出多轮循环回到 IDLE。

### 2.4 流式 LLM + 工具调用

每轮对话的 LLM 请求携带 `AVAILABLE_TOOLS`（5 个工具声明）。LLM 可返回两种结果：

**纯文本**：正常流式输出 → 句子分割 → TTS 播放。

**工具调用**（`tool_calls`）：LLM 返回 `{text, tool_calls: [{function: {name, arguments}}]}` → 执行以下流程：

```
LLM 返回 tool_calls
    │
    ▼
ToolExecutor.execute(name, arguments)
    │
    ├─ 成功: tts.enqueue_sentence(result)  ← 立即语音反馈
    │        dialog.add_tool_interaction()  ← 注入对话历史
    │        LLM 二次请求（基于工具结果生成自然语言回复）
    │
    └─ 失败: result["success"]=False → 错误信息同样播放并回传 LLM
```

**关键实现细节**：
- 流式模式下工具调用数据分块到达（DashScope 返回原生 dict），`llm_client.py` 的 `_safe()` 函数需同时支持 `getattr()` 和 `obj[key]` 两种访问方式
- 二次 LLM 请求前需重置 `reply = ""` 和 `skip_detected = False`，避免闭包内类型错误

## 三、专注模式业务逻辑

### 3.1 状态机（M7：正式状态机已完成）

**M7 已实现正式状态机**。`ToolExecutor` 将状态管理委托给 `StateController`（`src/state_controller.py`），FocusState 枚举包含 6 个状态：

```
IDLE ──set_focus_mode──▶ WAITING_PHONE ──IR检测到──▶ FOCUSING
                              │                         │
                              │ 60s超时               open_phone_box(temporary)
                              ▼                         ▼
                          返回失败                    PAUSED
                                                        │
                                              IR检测手机放回
                                                        │
                                                        ▼
                                                     FOCUSING（恢复）
                                                        │
                                              end_focus_mode / 计时到期
                                                        │
                                                        ▼
                                                      IDLE
```

### 3.2 进入专注模式 (`set_focus_mode`)

**触发条件**：LLM 根据用户话语调用工具函数。系统提示词第13条明确要求：必须通过函数调用执行操作，绝不能仅用文字回复"已开启"来假装完成。

**执行流程**：
1. **冲突检测**：如果已在专注中 → 返回错误提示
2. **状态转移**：IDLE → WAITING_PHONE（等待放入手机）
3. **IR 检测与兜底**：
   - 传感器子进程独占 GPIO17，仅在 IR 状态变化时发送 `PHONE_DETECTED`/`PHONE_REMOVED` 消息
   - 主循环 `_check_sensor_messages()` 接收消息并调用 `phone_inserted()`
   - **关键兜底**：若启动时手机已在盒中（IR 上电即检测到），初始 `PHONE_DETECTED` 在 IDLE 期间被丢弃。`_last_phone_state` 追踪最近状态，在 WAITING_PHONE 时主动触发 `phone_inserted()`
   - IR 去抖：需连续 5 次（1000ms）一致才确认状态变化
4. **关盖**：左右舵机关闭至 `box_close_angle`（默认 90°）
5. **LED**：绿色常亮
6. **计时器启动**：`FocusTimer.start(duration * 60)`，独立 daemon 线程每秒倒计时
7. **摄像头跟踪启动**：`camera.start_tracking()` 自动初始化 picamera2，启动 PID 人脸跟踪 + 走神检测线程

**时长范围**：5-120 分钟，默认值在 LLM tool definition 中为 25 分钟。

### 3.3 暂停与恢复 (`open_phone_box`)

**触发条件**：用户说"暂停一下"、"接个电话"、"拿一下手机"等 → LLM 调用 `open_phone_box(reason="temporary")`。

**执行流程**：
1. **暂停计时**：`FocusTimer.pause()`
2. **停止跟踪**：`camera.stop_tracking()` 设置 `_running=False` 真正停止跟踪线程，云台回默认位置（pan=90°, tilt=80°）
3. **开盖**：舵机打开至 `box_open_angle`（默认 0°）
4. **LED**：黄色常亮
5. **TTS 播报**：状态机 `on_tts_speak` 回调播放确认语音

**两种触发路径**：
- LLM 工具调用 `open_phone_box(reason="temporary")` → `request_pause()` → 立即执行开盖
- IR 物理检测到手机取出 → `phone_removed()` → 仅当状态为 FOCUSING 时执行暂停

### 3.4 结束专注模式 (`end_focus_mode`)

**触发条件**：
- 用户说"结束专注"、"停止学习"、"今天就到这里"等 → LLM 调用 `end_focus_mode`
- 计时器到期 → main.py 检测 `timer_expired` 标志 → 自动调用 `end_focus_mode({"_auto_expired": True})`

**执行流程**：
1. **停止计时**：`FocusTimer.stop()`
2. **停止跟踪**：`camera.stop_tracking()`
3. **清除标志**：`focus_active = False`, `focus_paused = False`
4. **开盖**：舵机打开
5. **LED**：蓝色呼吸
6. **TTS**：
   - 到期自动："专注时间到！辛苦啦，起来活动一下吧。"
   - 用户主动："专注模式已结束，盒盖已打开。辛苦啦！"

### 3.5 冲突检测

专注模式进行中时，LLM 再次调用 `set_focus_mode` → 返回 `success=False`，消息为"当前专注模式还在进行中，还剩X分钟。请先结束当前专注模式，再说出你想要的新时长。"

### 3.6 计时器 (`FocusTimer`)

- **实现**：独立 daemon 线程，每秒 `remaining -= 1`
- **到期**：`remaining <= 0` 时设置 `_expired = True` 并调用 `on_expire` 回调
- **暂停**：`paused = True`，线程仍在运行但不递减
- **停止**：设置 `running = False`，线程退出
- **线程安全**：`remaining` 读写使用 `threading.Lock`

## 四、硬件设计及逻辑

### 4.1 系统硬件架构

```
Raspberry Pi 5
├── I2C Bus 1
│   ├── PCA9685 (0x40) ──┬── CH0: 手机盒左舵机 SG90 (0-180°)
│   │                    ├── CH1: 手机盒右舵机 SG90 (0-180°)
│   │                    ├── CH2: 云台水平舵机 SG90 (0-180°)
│   │                    └── CH3: 云台俯仰舵机 SG90 (0-140°)
│   └── VL53L0X (0x29) ──── TOF 距离传感器（坐姿检测）
├── MIPI CSI
│   └── OV5647 ──────────── 摄像头（人脸跟踪 + 走神检测）
├── GPIO
│   ├── GPIO 17 ──────────  IR 红外避障传感器（手机盒检测）
│   ├── GPIO 23 ──────────  RGB LED - R 通道
│   ├── GPIO 24 ──────────  RGB LED - G 通道
│   ├── GPIO 25 ──────────  RGB LED - B 通道
│   └── GPIO 27 ──────────  物理按钮（短按/长按）
└── USB
    ├── USB 麦克风
    └── USB 扬声器
```

### 4.2 舵机系统（PCA9685 + SG90 × 4）

**驱动方式**：PCA9685 16 通道 PWM 驱动板，通过 I2C（地址 0x40）通信，50Hz PWM 频率。

**角度映射**：SG90 舵机 0° = 500μs 脉宽，180° = 2500μs 脉宽。

| 舵机 | PCA9685 通道 | 限位 | 开盒/默认 | 关盒/跟踪 |
|------|-------------|------|----------|----------|
| 手机盒左 | CH0 | 0-180° | 0°（开） | 90°（关） |
| 手机盒右 | CH1 | 0-180° | 0°（开） | 90°（关） |
| 云台水平 | CH2 | 40-180° | 90°（默认） | PID 动态 |
| 云台俯仰 | CH3 | 0-140° | 80°（默认） | PID 动态 |

> 限位由 `servo.pan_angle_min/max` 和 `servo.tilt_angle_min/max` 配置，防止连接线拉扯。

**速度模拟**：`set_angle()` 含移动延时 `abs(delta) / 90` 秒（最大 2 秒），真实舵机需机械运动时间。

**Mock 模式**：`ServoMock` 实现相同接口，仅日志输出角度变化。

### 4.3 红外传感器（手机盒检测）

**型号**：IR 红外避障模块  
**接口**：GPIO 17，`pull_up=True`  
**逻辑**：遮挡 = 手机存在（`read() = True`），无遮挡 = 空盒（`read() = False`）

**Debounce 滤波**（`wait_for_phone` / `wait_for_phone_removed`）：
- 采样间隔 200ms
- 需连续 3 次采样状态一致才确认
- `wait_for_phone` 确认后额外等待 1.5 秒（防夹手）
- 超时时间可配置（放入 60s，取出 30s，放回 120s）

**Mock 模式**：`IRSensorMock` 默认 `_auto_detect=True`，等待方法 0.1 秒后立即模拟成功。可调用 `set_auto_detect(False)` 切换为真实轮询模拟以测试超时路径。

### 4.4 TOF 距离传感器（坐姿检测）

**型号**：VL53L0X  
**接口**：I2C（地址 0x29）  
**范围**：30-2000mm  
**当前状态**：驱动已就绪（`read_distance()` + `close()`）。坐姿持续监测已集成至传感器子进程（`src/processes/sensor_process.py`），通过 `MessageBus` 发送 `POSTURE_WARNING` 事件。

### 4.5 RGB LED（状态指示）

**接口**：GPIO 23(R)、24(G)、25(B)，PWM 控制。

| 专注状态 | 颜色 | 模式 |
|---------|------|------|
| 进行中 | 绿色 | 常亮 |
| 暂停中 | 黄色 | 常亮 |
| 已结束 | 蓝色 | 呼吸 |
| 非专注 | 关闭 | — |

### 4.6 摄像头系统（OV5647 + MediaPipe）

#### 人脸跟踪（`FaceTracker`）

**PID 控制**：位置式离散 PID，setpoint = 画面中心(320, 240)。

```
误差 = 人脸中心 - 画面中心
死区 = 40px(X) / 30px(Y)  ← 避免微小抖动
输出 = KP×误差 + KI×积分 + KD×微分
```

- 每帧调用 `tracker.update(face_rect, 640, 480)`
- 仅在误差超过死区时调整舵机角度
- 角度增量 = PID 输出 × 0.05（平滑系数）

**自动扫描**：人脸丢失超过 10 秒后，水平方向以 5°/200ms 循环扫描，到达边界反向。重新检测到人脸后立即停止扫描并恢复跟踪。

**默认位置**：无人脸时云台回到 pan=90° / tilt=45°。

#### 走神检测（`DistractionDetector`）

基于 MediaPipe Face Mesh（478 个面部关键点）：

**闭眼检测**：
- 计算 Eye Aspect Ratio（EAR）：`(垂直距离和) / (2 × 水平距离)`
- EAR < 0.2 持续 10 帧 → 判定为闭眼

**转头检测**：
- 6 点 PnP（鼻尖/下巴/眼角/嘴角）估算头部姿态
- |yaw| > 20° 或 |pitch| > 15° 持续 10 帧 → 判定为转头

**综合判断**：闭眼或转头任一触发 → `distracted = True` → 回调 `on_distracted(reason)`

#### 性能优化（树莓派 5）

| 操作 | 频率 | CPU负载 |
|------|------|---------|
| Haar 人脸检测 | 每 3 帧 | ~4-5% |
| MediaPipe 走神检测 | 每 5 帧 | ~6-10% |
| PID 计算 + 舵机指令 | 每帧 | <1% |

配置项：`face_detection_interval: 3`, `distraction_interval_frames: 5`

#### Mock 模式

`CameraMock` 运行模拟跟踪线程，提供 `simulate_face_lost/found/moved` 和 `simulate_eyes_closed/looking_away` 方法供测试。

### 4.7 物理按钮

**接口**：GPIO 27，上拉，50ms 去抖  
**逻辑**：
- 短按（< 1s）：可注册回调
- 长按（> 3s）：可注册回调  
**当前状态**：驱动已就绪，但主循环中尚未绑定具体功能。M7 将实现按钮唤醒/暂停等功能。

## 八、开发约定

- 导入：`from src.xxx import YYY`（绝对导入）
- 日志：`logger = setup_logger("name")` 模块级
- 配置：所有可调值通过 `config.get("key.path", default)` 读取，禁止硬编码
- Mock/Real：通过 `devices/__init__.py` 工厂函数切换，禁止在业务代码中判断 `isinstance`
- 真实驱动：硬件库导入放在 `__init__` 内部（懒加载），确保 PC 可导入
